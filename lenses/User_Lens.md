# User Lens

## Lens Name
User Lens

## Purpose
Evaluate how the decision affects the end user, internal customer, consuming team, usability, friction, experience, and actual value delivery. This lens keeps the tool honest about whether the decision improves life for the people who feel its consequences.

## When to Use
- When the proposal changes workflows, interfaces, adoption paths, or the daily experience of a customer or consuming team.
- When the initiative is internally motivated and needs to be checked against actual user value.
- When a platform or organizational decision could easily optimize for internal convenience over external usefulness.

## Key Questions
- Who is the primary user or consuming team, and what do they actually feel today?
- What gets simpler, faster, clearer, or more reliable for them if this works?
- What new friction, migration cost, or learning curve might the proposal introduce?
- Does this solve a real user problem or mainly an internal proxy problem?
- Why would the user adopt the new path rather than stick with the current one?

## What This Lens Looks For
- Real experience improvements versus internal abstraction wins that never become user value.
- Friction in onboarding, migration, understanding, and day-to-day usage.
- Mismatches between what builders value and what users actually notice.
- Places where the proposal assumes adoption instead of earning it.
- Internal customer pain that is easy to dismiss because it is spread across many teams.

## Common Risks / Failure Modes
- Solving for internal neatness while making the user journey more complex.
- Requiring users to absorb migration pain before they feel any payoff.
- Assuming users share the organization's internal priorities or architecture language.
- Measuring success with internal proxy metrics while real user friction remains unchanged.
- Treating adoption as mandatory and therefore not needing to be designed for.

## Signals or Evidence to Consider
- User feedback, support themes, and workflow pain points.
- Current adoption behavior and reasons teams choose or avoid similar paths.
- Where the proposal removes friction users actually complain about.
- Time-to-value for the consuming team.
- Friction introduced by migration, training, or policy change.

## Output Expectations
- Describe the expected user or consumer experience if the decision proceeds.
- Surface where the proposal may create new friction even while solving internal problems.
- Call out whether the user value is direct, indirect, speculative, or delayed.
- Recommend how to make the new path easier to understand and adopt.

## Recommendation Style
Stay close to lived experience. Recommendations should favor concrete user impact over abstract strategic rhetoric.
